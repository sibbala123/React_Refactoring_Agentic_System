The Problem
Why is Automated React Refactoring Hard?
1
Silent Behavioral Regressions
Dropping a React context provider passes TypeScript but silently breaks form validation at runtime — invisible to static analysis.
2
State & Side-Effect Complexity
Hooks, useEffect dependencies, memoization — moving code between components can introduce stale closures and referential equality bugs.
3
Component Interdependence
Props, shared context, and rendering hierarchies create tight coupling. Changing a prop surface must be consistent with every call site.
4
Verification Limitations
TypeScript + linting are necessary but insufficient. Passing static checks ≠ runtime correctness. Most projects lack adequate test coverage.
Key Insight
We tested GPT-4o-mini and GPT-4o directly on real Supabase components with neutral refactoring prompts.

Both models dropped the <Form_Shadcn_> context provider — a hallucination invisible to TypeScript that silently breaks all form validation at runtime.

The answer to "can an LLM refactor React code reliably with a generic prompt?" is no.
Conclusion: Single-shot LLM refactoring is unsafe. We need structured coordination: classification, planning constraints, verification gates, and self-correction.
Foundation
Related Works
State of the Art
Recent work spans AST-based React smell detection (ReactSniffer, IST 2023), TypeScript/TSX extensions (IST 2025), AST vs AI-based correction comparisons (SSRN 2025), and MANTRA-style multi-agent refactoring systems for end-to-end automated repair.
Contextual Gap
While detection is mature, reliable automated remediation at scale remains challenging due to framework-specific structural invariants that general-purpose LLMs often violate.
The Research Frontier
Current tools either rely on rigid AST transforms (low flexibility) or raw LLM prompts (low reliability). ReactRefactor bridge this by using Agentic Workflows to apply AST-validated framework maneuvers.
Focus: Moving from "Detecting technical debt" to "Reliable, autonomous debt retirement" in complex frontend ecosystems.

Conclusion
Key Takeaways
What Works
Isolated Smells: Near-perfect acceptance for standalone components (Navbar/Dashboard).
Direct Tactics: add_controlled_state achieves 100% success with zero retries.
The Challenge
Entanglement: Multi-component files (Matches.jsx) cause surgical split conflicts.
Dirty State: Pre-existing uncommitted changes break patch verification gates.
Surprising Insights
Failures: Often path hallucinations (truncated directory names) rather than logic complexity.
Latency: 10x deviation in large files (AuditLogs.tsx) due to context pressure.
Outliers & Triage
Implicit Triage: 4 smell categories (Force Update, DOM Manip) were systematically untouched.
Invalid Plans: Plan-level errors remain unrecoverable by the standard retry loop.
Lessons Learned
Validation: Path normalization must precede scope validation.
Strategy: Reliability is high for self-selected subsets; full-scale coordination is the next frontier.
Conclusion
Scalability proven for isolated tasks ($0.005/refactor). Entangled files and global coordination are the primary failure frontiers.
